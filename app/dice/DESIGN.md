Этот документ — живой референс по фичам парсера. Для каждой фичи указан текущий статус,
уровень сложности реализации и комментарии по синтаксису или архитектуре.

Принципы, которыми руководствуемся при проектировании синтаксиса: см. `/AGENTS.MD`.
Ключевое: частые операции — короткий синтаксис; редкие — длиннее, но читаемые.

Сложность: **легко** / **средне** / **сложно** / **архитектура** (требует решений вне парсера).

## 1. Типы данных

**Статус:** ✅ реализовано (`RollValue` → `ScalarResult` / `ArrayResult`)

Текущая архитектура: каждое выражение возвращает скалярный `ScalarResult` (одно число + трейс).
Это работает для арифметики и кубов, но ломается при добавлении Repeat и Map — они возвращают
**список** результатов.

**Переработку нужно делать первой из сложных фич** — пока реализовано мало, переделывать дёшево.

### Предлагаемая иерархия

```
RollValue (базовый)
  ├── ScalarResult      # текущий RollResult — одно число, трейс
  └── ArrayResult       # список ScalarResult (от repeat/map)
        total: int      # сумма (для использования в арифметике)
        items: list[ScalarResult]
```

`ArrayResult` в арифметическом контексте ведёт себя как своя сумма:
`sum(5x(d6))` = то же что `5x(d6)` в позиции числа.

**Открытый вопрос:** может ли `map` возвращать `Array[Array]`? Например, `map(3x(d6), ...)` —
каждый элемент уже массив. Скорее всего, flatten на уровне map, но стоит обсудить.

**Сложность:** сложно (затрагивает evaluator, types, форматтер)

## 2. Базовые кубы

**Статус:** ✅ реализовано (`d`, `д`, `Nd`)

### `d%` — процентный куб

**Статус:** ❌ не реализовано
**Сложность:** легко — alias `d%` → `d100` на уровне лексера

```
d%     → бросить d100
3d%    → бросить 3d100
```

## 3. Преимущество и помеха

**Статус:** ✅ реализовано (`ad`, `dd` / `ад`, `дд`)
**Статус `ed`:** ❌ не реализовано

Логика длины синтаксиса соответствует частоте использования:

| Синтаксис | Кубов | Название | Частота |
|-----------|-------|----------|---------|
| `ad20`    | 2     | преимущество | часто |
| `dd20`    | 2     | помеха | часто |
| `ed20`    | 3     | тройное преимущество (Elven Accuracy) | редко |

**Сложность:** легко — `ed`/`kd` реализуются по шаблону `ad`/`dd`

**Вопрос из DESIGN_QUESTIONS:** альтернативные синтаксисы `d20adv` / `adv(d20)` — не нужны,
текущий `ad20` лаконичнее и единообразнее.

## 4. Взрывающиеся кубы

**Статус:** ❌ не реализовано

Взрывающийся куб: при выпадении максимума — перебросить и добавить к сумме, рекурсивно.

### Синтаксис

Референс предлагает `b6` (отдельный тип куба). Альтернатива — постфиксный `!`:

```
d6!    → взрывающийся d6 (взрыв при 6)
d8!    → взрывающийся d8 (взрыв при 8)
b6     → то же (кириллически: б6)
```

Оба варианта читаемы; `d6!` привычнее тем, кто знаком с другими системами, `b6` короче.
**Можно поддержать оба** — они не конфликтуют.

Взрыв при конкретном значении (не только максимуме):
```
d6!5   → взрыв при 5 и 6
d6!>=4 → взрыв при 4, 5, 6
```

**Открытые вопросы:**
- Лимит рекурсии — максимум N перебросов (например, 100). Без лимита — потенциальный бесконечный цикл.
- Нужен ли взрыв при конкретном значении (не только max)? Полезно для некоторых систем.

**Сложность:** средне — рекурсия в evaluator, нужен лимит

## 5. Модификаторы кубов

### `p` — pick best (оставить лучшие)

**Статус:** 🟡 частично реализовано

✅ Pick-best: `4d6p3` — бросить 4d6, взять 3 лучших. Кириллица: `4д6п3`.
```
4d6p3    → бросить 4d6, взять 3 лучших (генерация характеристик D&D)
5d20p2   → взять 2 лучших из 5
```

❌ Pick-worst — не реализовано:
```
4d6pw3   → взять 3 худших
```

**Сложность:** средне — новый тип узла в грамматике и evaluator

---

### Минимальное значение куба

**Статус:** ❌ не реализовано
**Синтаксис:** ⚠️ требует решения

Текущий синтаксис из старого референса: `d20%10` — конфликтует с `d%` (процентный куб)
и потенциально с оператором modulo.

Варианты:
- `d20lo10` — "lower limit 10", читаемо, нет конфликтов
- `d20m10` — коротко, но `m` неочевидно без контекста
- `d20^10` — `^` свободен, визуально "поднять до уровня"
- `d20[min:10]` — избыточно для этой операции
- Оставить `d20%10`, решить конфликт граматически — `%` после числа = modulo, `%<digit>` после куба = минимум

**Рекомендация:** `d20lo10`, потому что `lo` = lower limit читается без пояснений.
Редкая механика (Reliable Talent) — длина синтаксиса оправдана.

**Сложность:** средне

## 6. Арифметика

**Статус:** ✅ реализовано (`+`, `-`, `*`, `/`, скобки, унарный минус)

Modulo `%` — **не реализовывать**, вступает в конфликт с синтаксисом кубов и редко нужен в TTRPG.

Целочисленное деление `//` — **можно добавить**, иногда нужно для округления вниз.
Сложность: легко.

## 7. Сравнения

**Статус:** ❌ не реализовано

Возвращают `1` (истина) или `0` (ложь). Нужны для success-counting систем и тернарного оператора.

```
d20 > 10          → 1 или 0
d20 + 5 >= 15     → проверка броска атаки
```

**Сложность:** средне — новые правила грамматики, новые узлы evaluator

## 8. Функции min / max / sum

**Статус:** ❌ не реализовано

```
min(d20+4, d20+8)      → меньший из двух бросков
max(b8, b6)            → Savage Worlds: куб навыка vs дикий куб
sum(5x(d6))            → сумма массива (нужен ArrayResult)
```

`min`/`max` работают на скалярах, реализуются без ArrayResult.
`sum` требует ArrayResult.

**Сложность:** средне для `min`/`max`; `sum` — после переработки типов

## 9. Repeat (повтор)

**Статус:** ❌ не реализовано
**Требует:** переработку типов данных (ArrayResult)

```
5x(d6)          → бросить d6 пять раз → [3, 1, 5, 6, 2]
6x(4d6p3)       → генерация 6 характеристик
10x(ad20 + 3)   → 10 бросков с преимуществом
```

Результат — `ArrayResult`. В арифметическом контексте (если снаружи нет map/sum) —
автоматически суммируется или отображается как список, в зависимости от контекста вывода.

**Открытые вопросы:**
- Вложенные repeat: `2x(3x(d6))`? Логично разрешить, но нужен `Array[Array]` или flatten.
  Скорее всего flatten — пользователь редко ожидает матрицу.
- Кириллический эквивалент: `5х(d6)` уже описан в референсе — оставить.
- Использование без скобок: может ли система распознать `5x d20+3` ?

**Сложность:** сложно (зависит от переработки типов)

## 10. Map (применение к массиву)

**Статус:** ❌ не реализовано
**Требует:** ArrayResult + Repeat

### Проблема старого синтаксиса

```
map(it * 2 : 5x(d6))    ← старый синтаксис
```

Проблемы:
1. `it` — не очевидно, что это "текущий элемент"
2. Три уровня вложенности только для простого преобразования
3. Двоеточие как разделитель внутри скобок — неожиданно

### Варианты нового синтаксиса

**Вариант A — for-in (декларативный):**
```
for x in 5x(d6): x * 2
```
Читается вслух естественно. `x` — явная переменная. Двоеточие отделяет тело цикла, как в Python.

**Вариант B — pipe (функциональный):**
```
5x(d6) | x -> x * 2
```
Pipe-оператор `|` + лямбда. Цепочки: `5x(d6) | x -> x*2 | x -> x+1`.

**Вариант C — map с явной лямбдой (из DESIGN_QUESTIONS):**
```
map(5x(d6), x -> x + 1)
```
Явное имя функции, понятная структура. Но `map` как слово длиннее.

**Рекомендация:** Вариант A (`for x in ...: expr`) — самый читаемый для нетехнической аудитории,
при этом достаточно гибкий. Вариант B элегантнее для цепочек, но `->` в Discord-чате может
быть неудобно набирать.

**Открытый вопрос:** нужна ли фильтрация (`filter`)? Или достаточно `sum(for x in ...: x > 3)`?

**Сложность:** сложно (symbol table для переменной `x`, зависит от типов и Repeat)

## 11. Условный оператор (тернарный)

**Статус:** ❌ не реализовано

```
d20 = 20 ? 4d6 + 5 : 2d6 + 5
```

**Нужен ли?** С синтаксисом `for x in ...: expr` тернарный по-прежнему полезен для
условного выбора без цикла — например, "если крит, то двойной урон".

Синтаксис `? :` привычен, но в Discord-тексте вопросительный знак читается как вопрос.
Альтернативы: `if cond then a else b` — многословно, но однозначно.

**Рекомендация:** Оставить `? :`, реализовать после сравнений. Редко нужен без макросов.

**Сложность:** средне (требует сравнений)

## 12. Аннотации (теги урона)

**Статус:** ❌ не реализовано
**Синтаксис:** ⚠️ требует решения

Старый подход — trailing word без скобок:
```
2d6 slashing + 1d8 fire     ← проблема: пробелы в теге невозможны, цифры ломают парсер
```

Лучше — квадратные скобки:
```
2d6 [slashing] + 1d8 [fire]
2d6 [sneak attack] + 1d8 [fire]    ← пробелы работают
```

Одинаковые теги группируются и суммируются в ответе:
```
2d6 [fire] + 1d4 [fire]  →  "12 fire"
```

**Сложность:** средне — лексер, evaluator, форматтер

## 13. Несколько выражений

**Статус:** ❌ не реализовано

```
d20 + 7, 2d6 + 3              → атака и урон отдельно
ad20 + 5, dd20 + 2            → два независимых броска
```

Каждое выражение через запятую вычисляется и отображается отдельно.

**Сложность:** легко — обёртка на уровне парсера, не затрагивает evaluator

---

## 14. Кириллические эквиваленты

**Статус:** ✅ реализовано (`д`, `ад`, `дд`)
**Не реализовано:** `ед`, `кд`, `б`, `п`, `х`

Кириллика — уникальная фича для русскоязычной аудитории. Реализуется через aliases в лексере,
не требует изменений в грамматике или evaluator.

**Сложность:** легко — добавить токены в лексер по мере реализации фич

## 15. Макросы и переменные

**Статус:** 🔲 не проектировалось
**Требует:** архитектурных решений вне парсера

Именованные выражения, которые можно переиспользовать:
```
/macro set attack "ad20 + 7"
/r attack
```

**Открытые вопросы (из DESIGN_QUESTIONS):**
- Хранение: в памяти бота (теряются при рестарте) или в БД (на пользователя / на сервер)?
- Нужен ли контекст при вычислении — передавать переменные в выражение?
  (`/r attack mod=5` → подставляет `5` в выражение)
- Совместимость с Map: если в выражении есть переменная `x`, она не должна конфликтовать
  с параметрами макроса.

**Это отдельная система, не часть парсера как такового.** Парсер должен поддерживать
передачу внешних переменных (через env/context), остальное — архитектура бота.

**Сложность:** архитектура (требует отдельного проектирования)
